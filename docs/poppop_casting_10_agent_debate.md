# PopPop言語「Casting / Flavor (Type Handling & Casting)」に関する10人委員会 議事録

## 【議題】
PopPop言語（右方向データフロー `>>`, `>!>` を特徴とする言語）における「キャストと型の扱い（Casting / Flavor）」の洗練。
以下の5つの論点を中心に、10名の専門家による徹底討論を実施し、最終仕様を策定する。

1. キャスト/型チェックの厳格さ（厳密な静的型、グラデュアルタイピング、ソフトなFlavorマッチングなど）
2. キャスト失敗時の挙動（Burnt Kernelとなり `>!>` へ流れるべきか？）
3. 右方向フローにおけるキャスト構文（例: `data >> as Int`, `data as Int`, `data :: Int`, `data >> Flavor(Int)`）
4. カスタムデータ形状（構造体/レコード/Flavor）のキャスト・暗黙的変換の扱い
5. PopPopの「優雅さと品格」をどう維持するか

---

1. **構文純粋主義者 (Syntax Purist)**: 視覚的な美しさと構文の一貫性を最重視する。
2. **型システム理論家 (Type System Theorist)**: 型の健全性、静的解析、グラデュアルタイピングを重んじる。

5. **ゲーム開発者 (Game Dev)**: 実行時オーバーヘッドのないゼロコストのキャストを求める。
6. **バックエンドアーキテクト (Backend Architect)**: 強牢なエラーハンドリングとパイプラインの停止機構を重視。

## 【議論録】

**モデレーター (あなた)**: 本日はPopPop言語の型キャスト・Flavor機能について議論します。PopPopは「データが右へ流れる」という基本哲学があります。まず、**論点3の「構文」**から入りましょう。右方向フローにおいて、どのような記述が最も美しいでしょうか？

**構文純粋主義者**: PopPopの核心は `>>` というストリームです。操作はすべてパイプであるべきです。したがって `data as Int` よりも、パイプラインの中間ステップとして `data >> Flavor(Int)` または `data >> to(Int)` のように書くべきです。

**教育者**: `data as Int` は他の言語（TypeScriptなど）に慣れた人には直感的ですが、PopPopの「流れる」メンタルモデルには `data >> as(Int)` の方が合致しますね。ただ、型の概念を「Flavor（味付け）」と呼ぶ世界観なら `data >> Flavor(Int)` は素晴らしい。

**コンパイラエンジニア**: 構文解析の観点から言うと、`>>` の右辺に特殊なキーワードを置くより、`Flavor()` を第一級の関数・マクロとして扱う `data >> Flavor(Int)` はASTの構築が非常にシンプルになります。

**言語哲学者**: PopPopの哲学において、データは「カーネル（種）」であり、それに「Flavor（味）」をコーティングして次の工程へ送る、というメタファーは非常に優雅です。構文は `data >> Flavor(Int)` で決まりでしょう。

**型システム理論家**: 構文はそれに同意します。では**論点1と4「厳格さとカスタムデータ」**に移りましょう。単なる Int への変換ならともかく、構造体（カスタムFlavor）へのキャストはどうあるべきか。私は「グラデュアルタイピング（段階的型付け）」と「構造的サブタイピング」の導入を提案します。

**UI開発者**: 賛成です！JSONのパースなどで、 `{ "name": "Pop", "age": 2 } >> Flavor(User)` のように流した時、キーが一致していればソフトにマッチ（Soft Flavor Matching）してほしい。

**セキュリティスペシャリスト**: 待ってください。Soft Matchingは危険です。予期せぬ余分なフィールドが存在した場合、あるいは暗黙の型強制（文字列の "2" が数値の 2 になるなど）が連鎖すると、インジェクションの温床になります。明示的なバリデーションを伴うべきです。

**実践的データエンジニア**: しかし実務では、少しの型のズレでパイプラインが全停止するのは困ります。 "100" という文字列が流れてきたら、`Flavor(Int)` は適度に空気を読んで数値にしてほしい。

**バックエンドアーキテクト**: そこで**論点2の「エラー処理（Burnt Kernel）」**が活きます。パースやキャストに失敗した場合、システムクラッシュ（例外）を起こすのではなく、「Burnt Kernel（焦げた種）」という特別なエラー状態値になり、それがそのままパイプを流れるべきです。

**言語哲学者**: 「焦げたポップコーンは弾けない」——美しいですね。つまり、`data >> Flavor(Int)` が失敗すると `BurntKernel` が生成され、通常の `>>` パイプではスキップされるかエラーとして伝播する。

**バックエンドアーキテクト**: その通り。そしてエラー処理専用の右方向パイプ `>!>` でキャッチします。
`data >> Flavor(User) >!> log_error >> save_db` という形です。

**ゲーム開発者**: ちょっと待って。毎回のキャストでエラーオブジェクト（Burnt Kernel）をヒープにアロケーションしてたらパフォーマンスが死ぬ。C++の `reinterpret_cast` のような、絶対に失敗しない（失敗したら未定義動作で構わない）「強制キャスト」も必要だ。

**コンパイラエンジニア**: その課題は構文で解決できます。安全なキャスト（実行時チェックあり）は `>> Flavor(T)` とし、エラー時はBurnt Kernelを返す。一方、ゼロコストの強制キャスト（コンパイラへの型アサーション）は `>> Trust(T)` または `>> !Flavor(T)` とするのはどうでしょう。

**型システム理論家**: `Trust(T)` 構文は健全性を破壊しますが、システムプログラミング用途の抜け道としては妥当です。

**モデレーター**: 素晴らしい議論です。まとめると、キャストは「データの味付け（Flavoring）」としてパイプラインに組み込まれ、失敗時には優雅にBurnt Kernelとなって `>!>` パイプへ流れる。厳格さについては、安全なソフトマッチングを基本としつつ、エラーをパイプラインで明示的にハンドリングする方針ですね。

## 【最終コンセンサス仕様 (The PopPop Casting & Flavor Specification)】

10名の専門家による議論の結果、PopPopにおけるキャスト/型の扱いは以下の仕様に決定した。

### 1. キャストの基本構文 (Right-ward Flavoring)
型キャストはパイプラインの一部として記述され、データに「味（Flavor）」をコーティングするメタファーを用いる。
* **構文**: `data >> Flavor(TargetType)`
* **例**: `"123" >> Flavor(Int)`

### 2. キャスト失敗時の挙動 (The Burnt Kernel & Error Flow)
キャストやパースに失敗した場合、プログラムは例外でクラッシュしない。代わりに **`Burnt Kernel`**（エラー状態値）を生成する。
* `Burnt Kernel` は通常の `>>` パイプでは処理をスキップ（ショートサーキット）し、右へ右へと受け流される。
* 最終的に、エラー専用パイプ **`>!>`** によって捕捉され、優雅に処理される。
* **例**:
  ```poppop
  "abc" >> Flavor(Int) >!> (err -> print("Cast failed!")) >> do_something
  ```

### 3. 型チェックの厳格さと構造的マッチング (Gradual & Soft Matching)
* カスタムデータ（構造体/レコード）へのキャストでは、**Soft Flavor Matching** を採用する。
* `{ name: "Alice", id: "42" } >> Flavor(User)` のように流した際、フィールドが適合すれば受け入れる（ダックタイピング的挙動）。
* ただし、暗黙の型強制（文字列 "42" -> 数値 42 など）は `Flavor` 関数の仕様として安全に行われ、情報落ちや致命的な不一致がある場合のみ `Burnt Kernel` となる。

### 4. 強制/ゼロコストキャストの提供 (Unsafe Trust)
パフォーマンス（ゲーム開発等）や、開発者が型を100%保証できる場面のために、チェックを伴わないコンパイラレベルの強制キャストを提供する。
* **構文**: `data >> Trust(TargetType)`
* これに失敗した場合の挙動は未定義（クラッシュ）となるため、利用は制限される。

### 5. PopPopの優雅さと品格
「データがカーネルとして流れ、Flavorで味付けされ、失敗すれば焦げて（Burnt）別のレーン（`>!>`）に弾き出される」という一貫した世界観が維持される。これにより、関数型言語のモナド（Result/Either）の堅牢性を持ちながら、PopPop独自の直感的で遊び心のあるコード記述が可能となる。
