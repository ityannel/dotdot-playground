# 🎈 PopPop 言語 大長編公式仕様書 (The Definitive Guide)

# PopPop言語仕様書 - 第1部: 哲学とパラダイム

PopPopは「左から右へと流れる思考」をそのままコードにするためのプログラミング言語です。

**1. データフロー（Pipeline >>）**
すべての処理は左から右へ流れます。関数もマッチングも データ >> 処理. という一方向のパイプラインで完結し、視線の逆行を防ぎます。

**2. 遅延評価（Pop）**
データは末端に到達するまで評価されず、必要な瞬間に弾けて（Popして）結果を出します。

**3. 引数の束縛（Bundle &）**
関数呼び出しからカッコとカンマを排除し、複数の引数を & で束ねて渡します。ノイズを削ぎ落とした純粋な論理のみがコードに残ります。

# PopPop 言語仕様書 - 第2部 (Extender & Reviewer Expanded Edition)

本仕様書は、PopPopプログラミング言語におけるデータ型、パターンマッチング、反復処理、およびエラーハンドリングの基本構造と、言語処理系の挙動限界に及ぶ詳細なエッジケース、ならびにその解決策を定義します。

## 1. データ型 (Data Types)

PopPop言語は、メモリ安全性、直感的なセマンティクス、および決定論的な動作を保証するために厳格な型システムを提供します。

### 1.1 基本型
*   **数値型 (Number)**
    *   **整数 (Int):** 64ビット符号付き整数 (2の補数表現)。範囲は `-9,223,372,036,854,775,808` から `9,223,372,036,854,775,807`。
        *   *例:* `42`, `-100`, `1_000_000` (アンダースコアは可読性のために無視されます)
        *   *エッジケース (オーバーフロー/アンダーフロー):* 加算や乗算で64ビットの範囲を超えた場合、デフォルトでは **パニック (プログラムの即時終了)** が発生します。
        *   *解決策:* 意図的なラップアラウンドが必要な場合は、組み込み関数 `add_wrap(a, b)` を使用するか、任意精度整数型 `BigInt` (標準ライブラリ) を利用します。
    *   **浮動小数点数 (Float):** IEEE 754 倍精度 (64ビット) 浮動小数点数。
        *   *例:* `3.14`, `-0.001`, `1.5e10`
        *   *エッジケース (ゼロ除算):* `Int` 型でのゼロ除算はパニックを引き起こしますが、`Float` 型でのゼロ除算 (`1.0 / 0.0`) はIEEE 754に準拠し `Infinity` または `-Infinity` を返します。`0.0 / 0.0` は `NaN` (Not a Number) になります。
        *   *エッジケース (NaNの比較と判定):* `NaN == NaN` は常に `false` を返します。パターンマッチングにおいても NaN はリテラルマッチしません。
        *   *解決策:* 値が NaN や Infinity になっていないか検証するため、`math.is_nan(x)` や `math.is_finite(x)` などの専用関数を使用する必要があります。また、IEEE 754準拠のため `0.0 == -0.0` は `true` となる点にも注意してください。
*   **文字列型 (String)**
    *   UTF-8エンコードされた不変 (immutable) なテキストデータ。
    *   *例:* `"Hello"`, `"こんにちは"`, `"PopPop\n"`
    *   *文字列補間:* `"${変数名}"` だけでなく、`"${式 + 1}"` などの任意の式展開がサポートされます。
    *   *エッジケース (不正なバイト列):* 外部ファイルやネットワークから不正なUTF-8バイト列を読み込んだ場合、文字列への変換関数は `err(InvalidUTF8)` (Result型) を返します。
    *   *エッジケース (インデックスアクセス):* `str[i]` のような直接のインデックスアクセスは、O(N)の計算量を避けるため、また文字境界の誤謬（サロゲートペアや書記素クラスターの分断）を防ぐため **言語仕様として禁止されています**。
    *   *解決策:* 文字列の一部を取得する場合は `str.chars()` (Unicodeスカラ値) や `str.graphemes()` (書記素クラスター) を介して走査します。バイナリデータとしてのO(1)アクセスが必要な場合は `str.bytes()` を使用してバイト列として扱います。
*   **真偽値型 (Boolean)**
    *   `true` または `false`。
    *   *論理演算の短絡評価 (Short-circuiting):* `A and B`, `A or B` において、Aの評価結果だけで全体の結果が確定する場合、Bは評価されません。Bが副作用を持つ関数の場合は実行されない点に注意が必要です。

### 1.2 複合型
*   **リスト型 (List)**
    *   順序付けられた要素の可変 (ヒープアロケート) 配列。
    *   *例:* `[1, 2, 3]`, `["a", "b"]`, `[]`
    *   *エッジケース (範囲外アクセスとOption型のネスト):* 存在しないインデックスにアクセスした場合、PopPopはパニックせず、**Option型である `none` を返します**。アクセスが成功した場合は `some(value)` となります。
    *   *解決策（型の区別）:* もしリストの要素自体が `Option` 型であった場合（例：`[none, some(1)]`）、インデックス `0` へのアクセスは `some(none)` を返します。これにより、「インデックスが範囲外（`none`）」と「要素が存在し、その値がnone（`some(none)`）」を型システム上で厳密に区別できます。
    *   *エッジケース (メモリ枯渇):* リストの巨大化によってメモリ確保に失敗した場合、OSレイヤーからのOOMキルによりプロセスが終了します。
*   **マップ型 (Map)**
    *   キーと値のペアのハッシュテーブルコレクション。挿入順序は保持されません。
    *   *例:* `{"name": "Pop", "age": 5}`, `{}`
    *   *キーの制約:* 現在のバージョンではキーは `String` 型に限定されます。
    *   *エッジケース (キーの存在とOptionのネスト):* 存在しないキーへのアクセス `map["not_found"]` は `none` (Option型) を返します。リスト同様、格納されている値が `none` の場合は `some(none)` となり、キーの不在と区別されます。
    *   *解決策:* 値を確実に取り出す場合は `map["key"] ?? "default_value"` のようにNull合体演算子を併用するか、パターンマッチを使用します。

### 1.3 制御型
*   **Option型 (`some(T) | none`)**
    *   値の不在を型システムレベルで強制します。nullポインタ例外を根絶するための基盤です。
*   **Result型 (`ok(T) | err(E)`)**
    *   処理の成功と、失敗（例外の代替）を表現します。Eはエラーオブジェクトの型です。

## 2. パターンマッチ `check:` (Pattern Matching)

PopPopにおける条件分岐の中心的な役割を果たし、デストラクチャリングと型推論を同時に行います。

### 2.1 基本構文と詳細な使用例
```poppop
check: response {
    {"status": 200, "body": b} >> print("Success: ${b}"),
    {"status": 404, ...} >> print("Not found"),
    {"status": code, "body": b} if code >= 500 >> print("Server Error: ${code}"),
    _ >> print("Unknown response")
}
```

### 2.2 特殊なパターンマッチ機能
*   **ネストされた分解:** リストの中のマップ、マップの中のOptionなど、深くネストされた構造を1行で分解できます。
    *   `[some({"id": id}), ...rest] >> print(id)`
*   **OR パターン (`|`):** 複数の条件をまとめることができます。
    *   `"yes" | "y" | "true" >> return true`
*   **Rest パターン (`...`):** リストやマップの「残りの部分」を変数に束縛、または無視します。
    *   `{"name": n, ...others} >> print(others)` （`others` は `name` を除いた新しいMapになります）

### 2.3 エッジケースと解決策
*   **変数シャドウイングのエッジケース:** `check:` 外部で定義された変数 `x` と同じ名前をパターン内で使用 (`some(x) >> ...`) した場合、パターン内の `x` は新しい変数として外部の `x` をシャドウ（隠蔽）します。
    *   *解決策:* 外部の変数の「値」と一致するか判定したい場合は、ピン演算子 `^` を使用します。例: `some(^x) >> ...` (中身が外部変数 x の値と等価な場合のみマッチ)。
*   **マップパターンの動的キー照合:** マップのパターンマッチではリテラルキー (`"status"`) しか指定できません。動的なキーで照合することは文法上許可されません。
    *   *解決策:* 動的なキーの検査は、ガード節を用いて行います。 `map if map.has_key(dynamic_var) >> ...`
*   **網羅性チェック漏れによるコンパイルエラー:** 全ての列挙型(Enum)や型のバリアントをカバーしていない場合、ビルドが失敗します。
    *   *解決策:* あえて無視する場合は必ず `_ >> ()` (何もしないデフォルトケース) を記述して明示的にオプトアウトします。

## 3. イテレーション `do each:` / `loop:` (Iteration)

副作用を伴うループと、データ変換のための反復を整理します。

### 3.1 コレクションの走査 `do each:`
**動作原理:** 内部的には対象オブジェクトの `.iterator()` メソッドを呼び出し、要素が尽きるまで反復します。

*   **シャドーイングとスコープ:** `do each: item in list` の `item` は、ブロック内でのみ有効なイミュータブルな変数（各反復で再束縛される）です。
*   **エッジケース (ConcurrentModificationError):**
    *   *問題:* `do each:` のブロック内で、走査中のリストやマップに要素を追加・削除した場合、イテレータの整合性が崩壊します。PopPopはこれを検知し、実行時パニック (`ConcurrentModificationError`) を即座に発生させます。
    *   *解決策:* 要素を削除・フィルタリングしたい場合は、リスト内包表記ライクな関数型メソッド（例: `list.filter(|x| x > 0)`）を使用するか、走査するリストのコピー（クローン）に対してイテレーションを行います。 `do each: item in list.clone() { ... }`

### 3.2 無限・条件ループ `loop:`
**動作原理:** 評価式が `true` である限り、ブロックを再評価し続けます。

*   **値を返すループ (`break` の拡張):**
    PopPopの `loop:` は式（Expression）であるため、値を返すことができます。
    ```poppop
    let result = loop: {
        let data = fetch_data()
        if data.is_ready() {
            break data.value // ループを抜け、結果として data.value を返す
        }
        sleep(100)
    }
    ```
*   **エッジケース (永久ブロック・フリーズ):**
    *   *問題:* `loop:` ブロック内にI/O待ちや状態の更新がなく、条件も変化しない場合、スレッドがCPUを100%消費し続けます。
    *   *解決策:* PopPopのランタイムスケジューラは、バックグラウンドでプリエンプションを行いますが、単一スレッドの純粋な計算ループの場合はデッドロックの温床となります。意図的なビジーウェイトには必ず `yield()` などの明示的な処理譲渡関数を挿入することを推奨します。

## 4. エラーキャッチ `>~>` (Error Catching)

深くネストしたエラーハンドリングを平坦化し、可読性を高めるPopPop独自の演算子です。

### 4.1 詳細な挙動とセマンティクス
`let value = <operation_returning_result> >~> |err| { <fallback_expression> }`

*   **右辺の遅延評価:** `<operation>` が `ok` を返した場合、右辺のパイプおよびクロージャ `|err| { ... }` は **一切評価（実行）されません**。これはパフォーマンス上重要です。
*   **演算子のチェーン:** 複数の失敗しやすい操作を連鎖させる場合、`op1 >~> |e| { op2 } >~> |e| { op3 }` のようにチェーンすることが可能です。前の段階で `ok` が返れば後続の `>~>` ブロックは全てスキップされ、最終的な `value` にバインドされます。

### 4.2 高度な使用例と型の強制
```poppop
// データベースからユーザーを取得。失敗時は新規作成して返す。
// どちらのパスを通っても `User` 型が返ることが保証される。
let user = db.get_user(id) >~> |err| {
    log.error("Failed to fetch: ${err}")
    // errの型に応じて分岐も可能
    check: err {
        NotFound >> db.create_default_user(id),
        ConnectionLost >> panic("DB is down!")
    }
}
```

### 4.3 エッジケースと解決策
*   **エッジケース (型の不一致):**
    *   *問題:* 左辺の成功時の型が `Int` なのに、右辺のフォールバックブロックが `String` を返そうとした場合、変数 `value` の型がコンパイル時に決定できず、型エラーとなります。
    *   *解決策:* 両者の型を統一するか、明示的に直和型（Union型） `Int | String` を変数として宣言します（PopPopがUnion型をサポートしている場合）。または、すべてを文字列化して返すようにキャストします。
*   **エッジケース (非同期処理との組み合わせ):**
    *   *問題:* `<operation>` が非同期 (Promise / Future) を返す場合、`>~>` はどのように動作するのか？
    *   *解決策:* `>~>` 演算子は非同期コンテキストを透過的にサポートします。左辺が `await operation()` のように非同期である場合、右辺のフォールバックも非同期コンテキストで実行され、全体として `await` の結果として安全に評価されます。
*   **エッジケース (ハンドラからの早期リターン):**
    *   *問題:* フォールバックブロックから親関数自体を抜けたい場合。
    *   *解決策:* ブロック内で `return` 文を使用します。この場合、式の結果を変数に代入する処理はスキップされ、親関数から直ちにエラー（または別の値）が返却されます。
    ```poppop
    let data = fetch_api() >~> |err| {
        return err(err) // 呼び出し元へエラーをバブリングさせる明示的な記述
    }
    ```

# パート3: 関数、モジュール、および組み込み関数

## 関数定義 (`new`)
Dot Dotでは、新しい関数は`new`キーワードを使用し、その後に関数名、パラメーター、およびコードブロックを続けて定義します。

```
new greet(name) {
    "Hello, " + name >> Display
}
```

## モジュール (`Import`)
組み込み関数の`Import`を使用すると、他の`.pop`ファイルを現在の環境にインポートして実行できます。

```
"utils.pop" >> Import
```

## 5.1 PopPop (..) 言語ディープチュートリアル

**PopPop** 言語チュートリアルへようこそ！PopPopは、データフローを強く意識し、ボイラープレート（定型コード）を減らし、コードが左から右へと物語のように読めるように設計されています。

このディープダイブでは、PopPopの核となる哲学や独自の構文要素について解説し、定番のFizzBuzzやモダンなREST APIなどの実践的な例を構築します。

---

### コアコンセプト

#### パイプライン演算子 (`>>`)
PopPopでは、データはある操作から次の操作へとシームレスに流れます。パイプライン演算子 `>>` は、左側の結果を右側の入力として渡します。これにより、関数呼び出しの深いネストを排除できます。

```poppop
# 代わりに: print(filter(map(data, uppercase), is_valid))
data >> map(uppercase) >> filter(is_valid) >> print()
```

#### 暗黙の変数 (`@`)
パイプライン内で作業する際、小さく無名な操作を定義する必要がよくあります。PopPopでは、パイプラインの**現在のアイテム**を参照するための `@` を導入しており、明示的なラムダ変数宣言（`x >> x.name` など）の必要をなくしています。

```poppop
users >> filter(@.age >= 20) >> map(@.name) >> print()
```

#### 文字列補間 (`{::key}`)
PopPopは文字列補間に `{::variable}` を使用します。`::` を使うことで、標準的なブロックやオブジェクトの波括弧 `{}` とデータバインディングを明確に区別し、パーサーに優しく、即座に認識できるようにしています。

```poppop
name = "Alice"
greeting = "Hello, {::name}! You are processing item #{::@.id}."
```

---

### 例1: PopPopでのFizzBuzz

定番のFizzBuzzの実装で、これらの機能がどのように組み合わさるか見てみましょう。パイプラインと暗黙の変数が、ロジックをいかに宣言的で美しくしているかに注目してください。

```poppop
# 1から100までの範囲を生成し、それをmap関数にパイプする
1..100 >> map(
    if @ % 15 == 0: 
        "FizzBuzz"
    elif @ % 3 == 0: 
        "Fizz"
    elif @ % 5 == 0: 
        "Buzz"
    else: 
        @
) >> print()
```

**動作の仕組み:**
1. `1..100` は数値のストリームを生成します。
2. `>> map(...)` は各数値を1つずつ処理します。
3. `map` の内部では、`@` 記号は現在評価されている数値を暗黙的に参照します。
4. `map` の結果は文字列と数値のストリームになり、それが出力のために `print()` にパイプされます。

### 例2: REST APIの構築

PopPopのパイプラインアーキテクチャは、Webサーバーにおけるリクエスト/レスポンスのライフサイクルに最適です。架空の `web` モジュールを使って小さなREST APIを構築してみましょう。

```poppop
import web

# ポート8080でサーバーを初期化する
web.server(port=8080)
    # シンプルなGETエンドポイントを定義する
    >> route("GET", "/api/hello")
    >> handle(
        # @ は入力されたRequestオブジェクトを暗黙的に参照する
        response = {
            "status": "success",
            "message": "Hello, {::@.query.name | 'Guest'}!"
        }
        @ >> web.json(response)
    )
    
    # 要求された数値までのFizzBuzzを返すエンドポイントを定義する
    >> route("GET", "/api/fizzbuzz/{max}")
    >> handle(
        # リクエストから 'max' パラメータを抽出し、デフォルトを100とする
        limit = @.params.max | 100
        
        # PopPopの標準的なデータフローを使用してFizzBuzz配列を構築する
        result = 1..limit >> map(
            if @ % 15 == 0: "FizzBuzz"
            elif @ % 3 == 0: "Fizz"
            elif @ % 5 == 0: "Buzz"
            else: @
        ) >> collect()
        
        # JSONレスポンスを送信する
        @ >> web.json({ "data": result })
    )
    
    # 最後に、接続のリスニングを開始する
    >> start()
```

**APIの例からの重要なポイント:**
- **ルートのチェーン:** サーバーオブジェクトは `route()` と `handle()` の定義を通って簡単に流れます。
- **リクエストコンテキスト:** `handle()` ブロック内では、`@` はHTTPリクエストを表します。`@.query.name` や `@.params.max` に簡単にアクセスできます。
- **インラインのデフォルト値:** フォールバック値として使用される `|` 演算子に注目してください（`@.query.name | 'Guest'`）。
- **クリーンなJSON補間:** `{::@.query.name}` は、リクエストデータをレスポンス文字列に完璧に注入します。

## 5.2 Pyodideとブラウザ統合アーキテクチャ

プレイグラウンドのアーキテクチャは、Pyodide（WebAssemblyベースのブラウザ向けPythonディストリビューション）を活用して、Pythonベースの言語インタプリタを完全にクライアント側で実行します。これにより、コード評価のためのバックエンドサーバーが不要になります。

### 1. ブラウザ内の仮想ファイルシステム（VFS）のセットアップ
アプリケーションがロードされると、JavaScriptはPyodideを初期化し（`loadPyodide()`）、`pyodideInstance.FS` を使用してブラウザのメモリ内に仮想ファイルシステムを即座に構築します。
* `src/` ディレクトリを作成します。
* コアとなるPythonソースファイル（`lexer.py`、`parser.py`、`evaluator.py`、`ast_nodes.py` など）とメインのエントリーファイル（`poppop.py`）をWebサーバーから動的にフェッチします。
* `FS.writeFile()` を使用して、これらのスクリプトを直接PyodideのVFSに書き込みます。これにより、ブラウザのサンドボックス内にインタプリタのエコシステム全体が再現され、Pythonはそれらをネイティブモジュールとしてインポートできるようになります。

### 2. コード評価のワークフロー
ユーザーがプレイグラウンドのUIで「Run（実行）」をクリックしたとき:
* Aceエディタの内容は、一時ファイル（`playground.poppop`）としてVFSに書き込まれます。
* JavaScriptは `pyodideInstance.runPythonAsync()` を呼び出し、ブリッジ用のPythonスクリプトを実行します。
* このブリッジスクリプトはVFSから `playground.poppop` を読み込み、コアインタプリタを呼び出します（例: `await poppop.run_code(source)`）。

### 3. リアルタイムI/Oリダイレクト（Python ↔ JS ブリッジ）
ターミナルの出力が瞬時に感じられるように、Pyodideの相互運用モジュール（`import js`）を使用して、Pythonの標準出力を直接DOMにブリッジします。
* Pyodideの内部では、`sys.stdout` と `sys.stderr` はカスタムの `RealtimeOutput` クラスによってオーバーライドされます。
* 言語評価器が `print()` を呼び出すと、このカスタムクラスの `write()` メソッドに渡されます。
* `write()` メソッドはグローバルなJavaScript関数 `window.appendTerminalOutput(s)` を直接呼び出し、出力ストリームをリアルタイムでDOMのコンソールウィンドウ要素に即座に追加します。

### 4. ASTの抽出とMermaid.jsビジュアライザ
このアーキテクチャは、抽象構文木（AST）のライブ視覚化もサポートしています。
* Pythonのブリッジスクリプトは単にコードを実行するだけでなく、シリアライズ関数（`poppop.get_ast_json(source)`）を呼び出してソースをパースし、ASTのJSON表現を出力します。
* このJSON文字列は、Pyodideの境界を越えてJavaScriptレイヤーに返されます。
* フロントエンドはその後、カスタムのJavaScriptトラバーサル関数（`buildMermaidGraph()`）を実行し、ASTのJSONをMermaid.jsのダイアグラム定義にマッピングして、出力とともにパイプラインフローのグラフィックを即座にレンダリングします。

これにより、トークン化や評価に関するPythonの堅牢なロジックの恩恵を受けながら、モダンなWebアプリケーションのサンドボックス化された、非常に高速でインタラクティブなユーザー体験を提供できます。
