## 💡 20人が考案した 4つの対案（Proposals）

---

### 提案A: 【PascalCase vs camelCase 規則】（大文字＝関数、小文字＝変数）
> **「見た目の美しさを一切崩さず、命名規則で100%識別する」**

* **規則**:
  * **大文字始まり（PascalCase）**: 100% **「関数 / 動詞 (Verb)」**
    * 例: `Display`, `Map`, `Filter`, `Sort`, `Sum`, `Max`, `Fetch`
  * **小文字始まり（camelCase / snake_case）**: 100% **「変数 / 名詞 (Noun)」**
    * 例: `scores`, `total_revenue`, `user_id`, `item`
* **メリット**: 記号が一切増えず、コードが最も美しい。
* **リンター補強**: もし `Map(Track):` のように変数を大文字で書いた場合、コンパイラが `[LinterWarning] Variable 'Track' should start with lowercase 'track'` と即座に警告する。

---

### 提案B: 【シジル / 記号接頭辞 規則】（Sigil Notation）
> **「記号（Prefix）をつけて視覚的に100%絶対識別する」**

* **規則**:
  * 変数には必ず **`$`** または **`@`** をつける。（例: `$scores`, `$total`）
  * 関数はそのまま表記。（例: `Display`, `Max`）
* **コード例**:
  ```poppop
  $scores >> Max >> $max_score.
  ```
* **メリット**: 文脈に依存せず、どこにあっても100%変数と判別できる。
* **デメリット**: 記号が多くなり、PopPopの爽快で美しいコードがゴチャゴチャする。

---

### 提案C: 【位置による文脈決定 規則】（Positional Contextual Resolution）
> **「パイプラインの位置（右側か、終端か）で文法的に一義決定する」**

* **規則**:
  * **`>>` の直後に単独で来る単語**: 文脈上 **100% 関数（動詞）** とみなす。
  * **`>> 変数名.` のように終端の `.` の直前に来る単語**: 文脈上 **100% 変数バインド（代入）** とみなす。
* **コード例**:
  ```poppop
  raw_data >> Filter >> Sort >> result.
  //          ↑関数   ↑関数   ↑変数(終端.)
  ```
* **メリット**: 大文字小文字の制約すら不要。パイプラインの位置だけでコンパイラが100%誤認なく解釈できる。

---

### 提案D: 【カスタム関数宣言 `new` 登録 規則】（Explicit Verb Symbol Table）
> **「ユーザー定義関数は必ず `new` で宣言し、シンボルテーブルに登録する」**

* **規則**:
  * 自作関数をつくる時は必ず `new MyFunc = (x): ... ..` と宣言する。
  * コンパイラは「組み込み関数リスト」＋「`new` で定義されたユーザー関数リスト」を【動詞テーブル】として保持。それ以外はすべて【名詞（変数）】として扱う。
* **メリット**: 関数の密造を防ぎ、名前衝突が起きない。

20人の専門家が討論した結果、**提案A（大文字＝関数 / 小文字＝変数）と 提案C（位置による文脈解決）を融合した【二重防護システム】** を全会一致で採択いたしました！

---

### 📜 【決議文】「PopPopにおける関数と変数の品格ある分離規範」

1. **基本識別ルール（提案A + 提案Cの融合）**:
   * **関数（動詞）**: 原則として **大文字始まり（PascalCase）**（例: `Display`, `Fetch`, `Sort`）。
   * **変数（名詞）**: 原則として **小文字始まり（camelCase）**（例: `scores`, `item`, `user_id`）。
   * **位置による確定**: パイプライン `>>` の途中にある単語は「関数」、終端 `>> var.` にある単語は「変数」としてコンパイラが位置で確定補正する。

2. **大文字変数の禁止・リンター警告ガイド**:
   * `Map(Track):` のように変数を大文字で宣言しようとした場合、パーサーが直ちに以下の親切なリンターメッセージを表示する：
     ```text
     [LinterStyle] Variable names must start with a lowercase letter to distinguish them from functions.
     👉 Suggested fix: Change 'Track' to 'track' or 'item'.
     ```

3. **VS Code 構文ハイライト（Syntax Highlighting）の強化**:
   * 関数（`Display`, `Map`, `Sort`）は **黄色/緑色** で強調表示。
   * 変数（`scores`, `item`）は **青色/白色** で強調表示。
   * これにより、カッコがなくてもエディタ上で一目で識別可能にする。
